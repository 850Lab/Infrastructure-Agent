GAP A — Auth completeness

Need:

Need:

- admin can assign machine config
- admin can run bootstrap/setup for that client
- admin can mark client machine as ready
- onboarding for the client still happens for real
- this flow must be explicit and repeatable

GAP E — Billing / usage guardrails
Need:
- usage caps per client or per run
- configurable max limits for expensive operations
- prevent runaway enrichment/query generation
- basic usage visibility for operator
- clear warnings when limits are approached

GAP F — Support / retry / recovery
Need:
- partial run failures visible
- retry safe
- failed steps resumable or re-runnable

GAP G — Data ownership / portability
Need:
- export path for client data
- clear separation of client data
- cancellation/offboarding-safe data export
- no lock-in assumptions inside code

GAP H — Admin control plane
Need:

GAP I — Environment / release safety
Need:
- production-safe config checks

- rollback-safe changes
- no hardcoded local assumptions
- no secrets in logs

GAP J — Client onboarding tied to provisioning
Need:
- provisioning creates the machine shell
- onboarding still runs for the client user

- machine config is editable later by allowed roles only

GAP K — Permissions model

GAP L — Commercial packaging support
