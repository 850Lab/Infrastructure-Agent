==================================================
CORE CONCEPTUAL MODEL
==================================================

The new app should be structured around:

Company
  -> Target Roles / Decision Makers
  -> Contact Flows
  -> Attempts
  -> Outcomes
  -> Next Action
  -> Pipeline Progression
  -> Nurture / Reactivation

Everything must start from the company record.

Contacts exist in the context of a company.
Flows exist in the context of a company and/or a company-contact pair.
Next actions must always be generated from actual flow state.

==================================================
THE 5 CORE FLOWS THE APP MUST SUPPORT
==================================================

FLOW 1 — Gatekeeper Discovery Flow
Purpose:
- identify the right decision maker
- get name, title, extension, email, transfer, or routing clue

This flow is company-level first, before direct DM contact is confirmed.

Track repeated attempts such as:
- no answer
- receptionist answered
- voicemail/general mailbox
- refused / blocked
- transferred
- gave name
- gave title only
- gave direct extension
- gave email
- asked to send info
- promised to pass message

This flow should support 6+ attempts over time.

==================================================

FLOW 2 — Decision Maker Direct Call Flow
Purpose:
- reach the decision maker directly
- log voicemail/live answer/conversation
- move toward qualification or meeting

Track:
- direct call attempt count
- live answer
- voicemail left
- asked to call back later
- unavailable
- wrong person
- referral to other person
- not relevant
- interested
- conversation happened
- meeting requested
- follow-up date given

This flow should support 6+ attempts over time.

==================================================

FLOW 3 — Email Flow
Purpose:
- reinforce legitimacy
- maintain contact when phone doesn’t connect
- document relevance
- give additional touchpoints

Track:
- email found / not found
- email sequence step
- sent date/time
- opened if available
- clicked if available
- replied
- bounced
- paused
- completed
- not relevant
- interested
- follow-up needed

This flow should support 4–6 emails over time.

==================================================

FLOW 4 — LinkedIn Flow
Purpose:
- establish familiarity and identity
- support the phone/email outreach
- add another path to the same DM

Track:
- profile not found
- profile found
- viewed
- connection requested
- connected
- message sent
- responded
- no response
- follow-up sent

==================================================

FLOW 5 — Long-Term Nurture Flow
Purpose:
- keep viable but non-responsive or non-timing-ready companies alive
- recycle companies back into active outreach when appropriate

Track:
- nurture start date
- nurture reason
- next check-in date
- seasonal/project-based reactivation
- quarterly follow-up
- refinery/turnaround season check
- reactivated into active flow

==================================================
BUSINESS LOGIC / NEXT ACTION ENGINE
==================================================

This is critical.

Build or adapt logic so the app can recommend the next best action based on outcomes.

Examples:

If Gatekeeper outcome = "gave DM name"
-> create/update contact
-> start DM Call flow
-> recommend direct call attempt #1

If Gatekeeper outcome = "gave email"
-> create or update contact/email
-> activate Email flow
-> recommend Email Step 1

If Gatekeeper outcome = "message taken"
-> schedule follow-up gatekeeper call in X days

If DM call outcome = "voicemail left"
-> schedule DM call retry
-> optionally suggest email if email exists

If DM call outcome = "live answer - not relevant"
-> close active flows or move to nurture/closed lost based on logic

If DM call outcome = "interested"
-> raise priority
-> recommend follow-up / meeting / presentation stage

If no response after max attempts across major channels
-> move to Nurture with next reactivation date

This logic should be visible and deterministic, not hidden black-box behavior.
