# Texas Automation Systems — Lead Engine Command Center

## Overview
This project is a multi-tenant B2B lead generation and call management system designed to automate and streamline lead processes, manage call operations, and provide real-time insights via a "Command Center" dashboard. It offers real-time data visualization, AI-powered lead enrichment, dynamic playbook generation, and comprehensive analytics. The system aims to serve Gulf Coast industrial contractors initially, with future plans for broader industry expansion by automating and optimizing lead management and sales outreach.

## System Architecture
The system employs a micro-frontend-like structure with distinct frontend and backend separation, supporting a multi-tenant architecture with role-based access control and client data isolation.

### UI/UX Decisions
The frontend is a React application built with Shadcn UI and Framer Motion, featuring a modern design with a white background, dark text, and emerald green accents. The dashboard includes a three-column layout, a dynamic 8-node SVG "Neural Network" visualization (Pulse Reactor), and uses Server-Sent Events (SSE) for real-time data updates.

### Technical Implementations
The **backend** is developed with Express and TypeScript, while the **frontend** is an 11-page React application optimized for performance with virtualization and React Query caching. **PostgreSQL** stores webhook logs, user accounts, and client registry. **Authentication** is database-backed with token-based security. **Real-time communication** is handled by Server-Sent Events (SSE) with an in-memory EventBus. A daily **orchestrator** manages lead generation steps. **Data persistence** primarily uses Airtable with JSON file fallbacks. **KPI tracking** computes and caches performance metrics. A **Call Recording + Transcription Pipeline** processes audio, transcribes it using Whisper, and performs containment analysis via deterministic rules and GPT-4o. A **Playbook Feedback Loop** leverages analyzed call data to refine outreach playbooks. A **Briefing Engine** generates daily summaries and recommended actions.

The system incorporates a **Closed-Loop Sales Learning System** with four layers: an **Observation Engine** for structured call insights, an **Interpretation Engine** for classifying interactions and computing severity scores, a **Pattern Engine** for aggregating insights across calls, and an **Optimization Engine** for generating structured `Script_Patches`. This system feeds into a **Script Evolution Layer** which creates operator-facing recommendations and manages playbook versioning.

A **DM Decision Authority Learning Loop** learns which decision maker titles lead to successful deals, evolving DM targeting and query generation based on real outreach data. This includes **Win Tiers** for classifying deal progression, **DM Outcome Tracking**, a **DM Authority Learning Engine** for title effectiveness, and **Adaptive DM Fit** for adjusting DM scoring. **No Authority Detection** automatically identifies "wrong person" calls using rule-based and AI analysis. **Query Generation Performance Tracking** compares lead quality across ColdStart, QueryIntel, and WinPattern modes — tracking leads produced, DM coverage rate, positive call rate, and opportunity rate per mode. The analytics page displays a "Query Strategy Performance" comparison section with per-mode cards and progress bars, highlighting the best-performing strategy. **Time-Decay Weighting** (`server/time-weight.ts`) ensures recent signals influence the learning system more than old ones, using exponential decay with a configurable constant (default 60 days via `decay_constant` in IndustryConfig). Applied to DM authority learning (weighted outcome counts), query intelligence scoring (weighted attribution), and opportunity engine priority (weighted engagement/opportunity scores). The dashboard Intelligence section includes a "Signal Recency" card showing the ratio of recent vs historical signal strength. **Authority Trend Tracking** stores historical snapshots of DM title effectiveness in the `authority_trends` PostgreSQL table (client_id, title, snapshot_date, conversion_rate, sample_size). Snapshots are automatically taken after each daily run via `snapshotAuthorityTrends()` in `server/dm-authority-learning.ts`. The `GET /api/authority-trends` endpoint returns the time series. The Analytics page displays a "Decision Maker Effectiveness Over Time" SVG line chart with color-coded title series, grid lines, date axis labels, a legend, and an auto-generated insight comparing current vs previous top performers. **Cross-Client Learning** (`server/platform-insights.ts`) aggregates anonymized title performance data across all client machines into the `platform_insights` PostgreSQL table. Title buckets (Safety Manager, Superintendent, Project Manager, etc.) track conversion effectiveness, DM reach rate, and sample size per industry. The `getPlatformDMBoost()` function injects a bounded adjustment (-15 to +15) into DM fit scoring based on how a title bucket performs relative to the cross-client average. The admin dashboard displays a "Cross-Client Insights" section with a "Refresh Insights" button to trigger aggregation on demand. Time-decay weighting is applied to cross-client outcomes for recency bias.

**DM Status Classification** (`server/dm-status.ts`) evaluates every company's decision-maker data quality and assigns a `DM_Status` field in Airtable. The engine runs `evaluateDMStatus(company)` against each record, checking: authority mismatch (NoAuthority/wrong_person outcomes or miss count >= 2), missing website, no DM found, generic email prefixes (info@, admin@, etc.), weak title fit (below moderate threshold), missing email, missing phone, and strong outreach readiness. Status values: `DM_READY`, `DM_WEAK`, `NO_DM`, `NO_EMAIL`, `NO_PHONE`, `GENERIC_CONTACT`, `NO_WEBSITE`, `AUTHORITY_MISMATCH`, `RECOVERY_IN_PROGRESS`, `READY_FOR_OUTREACH`. The `updateDMStatus(clientId)` function fetches all companies, classifies each, and batch-updates Airtable. It auto-creates the `DM_Status` and `DM_Last_Checked` fields if they don't exist. Runs in the daily pipeline after DM Fit (`dm_status` step). Manual trigger: `POST /api/dm-status/run`.

**DM Recovery Queue** (`server/recovery-engine.ts`, `recovery_queue` PostgreSQL table) continuously reprocesses companies without usable DMs using a persistent queue with backoff scheduling. The `recovery_queue` table stores: `company_id`, `company_name`, `dm_status`, `priority`, `attempts`, `next_attempt`, `last_result`, `active`. Companies are added when `DM_Status` is in {NO_DM, NO_EMAIL, NO_PHONE, GENERIC_CONTACT, AUTHORITY_MISMATCH, NO_WEBSITE}. Priority: AUTHORITY_MISMATCH=1_highest, NO_DM/NO_WEBSITE=2_high, NO_EMAIL/NO_PHONE=3_medium, GENERIC_CONTACT=4_low. Each daily run: `populateRecoveryQueue()` syncs Airtable statuses into the queue (adds new, removes DM_READY), then `processRecoveryQueue()` fetches items where `next_attempt <= now` (up to 20 per run) and executes recovery strategies (re-enrichment, Outscraper lookup, email pattern, gatekeeper referral). Backoff: attempts 1-3 → +2 days, 4-6 → +7 days, >6 → +30 days. Items removed from queue when `DM_Status` becomes `DM_READY` or `READY_FOR_OUTREACH`. Recovery results written to Airtable fields: `Recovery_Plan`, `Recovery_Attempts`, `Recovery_Last_Run` (auto-created if missing). API: `POST /api/recovery/run` (trigger full cycle), `GET /api/recovery/queue` (list queue with stats). Runs in daily pipeline before opportunity engine.

**Information Ceiling Detection** (integrated into `server/recovery-engine.ts`) determines when the system has exhausted all automated recovery options for a company. After each recovery attempt in `processRecoveryQueue()`, if `Recovery_Attempts >= 6` AND no new DM or contact data was discovered, the company hits the information ceiling. When triggered: `Info_Ceiling_Reached` is set to `true`, `Info_Ceiling_Date` is set to the current timestamp, `DM_Status` is changed to `READY_FOR_OUTREACH`, and the company is deactivated in the recovery queue. Meaning: "this is all the information we can obtain automatically." Airtable fields `Info_Ceiling_Reached` (checkbox) and `Info_Ceiling_Date` (dateTime) are auto-created via `ensureRecoveryFields()` if they don't exist. The `processRecoveryQueue()` return type includes a `ceilingReached` counter. The threshold is configured via `INFO_CEILING_THRESHOLD = 6`.

**6-Touch Outreach Pipeline** (`server/outreach-engine.ts`, `outreach_pipeline` PostgreSQL table, `client/src/pages/active-outreach.tsx`) generates and manages structured outreach sequences for companies that reach `DM_READY` or `READY_FOR_OUTREACH` (including Info Ceiling). The pipeline creates a 6-touch sequence per company: Touch 1 (Email, Day 1), Touch 2 (Call, Day 3), Touch 3 (Email, Day 5), Touch 4 (Call, Day 7), Touch 5 (Email, Day 10), Touch 6 (Call, Day 14). Email content is generated via GPT-4o using the existing playbook engine patterns. Call scripts are generated deterministically based on touch number and DM info. The `outreach_pipeline` table stores: `company_id`, `company_name`, `contact_name`, `touch_1_email` through `touch_6_call`, `pipeline_status`, `next_touch_date`, `touches_completed`. Pipeline statuses: `ACTIVE`, `COMPLETED`, `RESPONDED`, `NOT_INTERESTED`. Each daily run: `populateOutreachPipeline()` finds eligible companies and generates sequences, `advanceOutreachPipeline()` progresses due items. API: `POST /api/outreach/run`, `GET /api/outreach/pipeline?status=ACTIVE`, `PATCH /api/outreach/pipeline/:id/status`. The "Active Outreach" page is accessible via dashboard nav and shows stats cards, status filter tabs, expandable outreach cards with touch timelines, and status action buttons. Runs in the daily pipeline after recovery engine (`outreach_engine` step).

**Machine Alerts** (`server/machine-alerts.ts`) is a proactive anomaly detection system that runs after each daily pipeline. It checks four alert types: `title_performance_change` (title conversions surging), `title_decline` (title effectiveness dropping), `authority_mismatch_spike` (high wrong-person/no-authority rates), and `query_performance_shift` (one query strategy significantly outperforming another). Alerts are stored in the `machine_alerts` PostgreSQL table (client_id, alert_type, message, severity, resolved, created_at). API endpoints: `GET /api/alerts?unresolved=true` and `POST /api/alerts/:id/resolve`. The Daily Briefing page (`client/src/pages/briefing.tsx`) shows a "Machine Alerts" section with severity-colored cards and dismiss buttons. The Command Center dashboard shows an alert count badge on the Daily Briefing button and displays a toast notification when unresolved alerts exist.

Key features include a **Command Center Dashboard**, **Lead Engine**, **DM Enrichment**, **Playbook Generator**, **Run Diff**, **Revert Last Run**, **Call Outcome Engine**, **Opportunities Pipeline**, **Call Mode**, **Machine Feedback**, **Rank Explainability Layer**, **Machine Identity & Settings**, **First-Run Cinematic**, **Onboarding Wizard**, and an **Admin Platform**. **Targeting Accuracy** is measured as a score based on data completeness for critical fields. The system includes **Auth & Client Context** management for multi-tenant operations and **Production Safety** measures such as React Error Boundaries, server Process Guards, Concurrency Guards, Rate Limit Handling, and Fetch Timeouts.
