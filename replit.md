# Texas Automation Systems — Lead Engine Command Center

## Overview
This project, "Lead Engine Command Center," is a multi-tenant B2B lead generation and call management system designed to automate and streamline lead processes, manage call operations, and provide real-time insights. Key capabilities include real-time data visualization, AI-powered lead enrichment, dynamic playbook generation, and comprehensive analytics. The system aims to optimize lead management and sales outreach, targeting industrial contractors with plans for broader industry expansion. It provides a central dashboard for monitoring and managing lead generation and sales activities.

## System Architecture
The system employs a micro-frontend-like architecture with distinct frontend and backend separation, supporting multi-tenancy with role-based access control and client data isolation.

### UI/UX Decisions
The frontend is a React application using Shadcn UI and Framer Motion, featuring a modern design with a white background, dark text, and emerald green accents. The dashboard includes a three-column layout, a dynamic 8-node SVG "Neural Network" visualization (Pulse Reactor), and uses Server-Sent Events (SSE) for real-time data updates. A public landing page/product tour is available at `/site` for an interactive guided demo.

### Technical Implementations
The **backend** is built with Express and TypeScript, while the **frontend** is an 11-page React application optimized with virtualization and React Query caching. **PostgreSQL** is used for various data, including webhook logs, user accounts, client registry, and system-specific tables. **Airtable** serves as the primary data persistence layer for business logic, with JSON file fallbacks. **Authentication** is database-backed and token-based. **Real-time communication** uses SSE with an in-memory EventBus. A daily orchestrator manages lead generation.

Key features include:
-   **Command Center Dashboard**: Central hub for operations.
-   **AI-Powered Engines**: Lead Engine, DM Enrichment, Playbook Generator, Call Outcome Engine, Briefing Engine, Closed-Loop Sales Learning System (Observation, Interpretation, Pattern, Optimization Engines), DM Decision Authority Learning Loop, Web Intel Scraper (for sales intel extraction), and Machine Alerts.
-   **Outreach Management**: Unified Outreach Hub for email and call execution, 6-Touch Outreach Pipeline (configurable for calls-first or email-first), Per-Client Email Sending & Tracking (with provider-aware limits and scheduled sending), Email Performance Analytics, Template Customization, and Reply Detection.

Shows action queue cards with flow-type labels, flow-specific outcome buttons (11 GK, 10 DM, 8 Email, 9 LinkedIn, 5 Nurture), gatekeeper intel capture (two-step: select outcome → enter info → confirm), scripts/intel drawer, session progress bar, and session summary screen. Backend: `server/flow-engine.ts` (deterministic Next Action Engine with `computeNextAction()`), `server/flow-routes.ts` (API). DB tables: `company_flows`, `flow_attempts`, `action_queue`. **Today Page** (`/machine/today`): Execution-first Action Dashboard with 7-column stat bar, flow-type filter tabs, overdue banner, "Activate Flows" button (seeds from Airtable today list via `seedFlowsFromTodayList`), "Start Focus Mode" button. **Airtable Write-Back** (`server/airtable-writeback.ts`): Every flow attempt syncs state back to Airtable Companies table — updates Lead_Status, Times_Called, DM_Status, Engagement_Score (clamped 0-100). Extended fields (Last_Touch_Date, Last_Touch_Channel, Last_Touch_Outcome, Next_Action, Next_Action_Due, per-flow status fields) are written only if they exist in the Airtable schema (safe fallback with field cache + core-only retry). Also syncs call records to Calls table for call-type flows. Fire-and-forget pattern (non-blocking). **State Integrity**: Duplicate flow prevention via `checkDuplicateFlow()` — blocks active/paused duplicates per company+flowType (DM flows additionally scoped by contactId). Max-attempt auto-nurture: when any non-nurture flow reaches max attempts with terminal status, auto-spawns a nurture flow. Queue sorting: overdue first, then highest priority, then earliest due date.

-   **Production Safety**: Includes React Error Boundaries, Process Guards, Concurrency Guards, Rate Limit Handling, and Fetch Timeouts.

## External Dependencies
-   **Airtable**: Primary data store.
-   **OpenAI**: Used for Whisper (audio transcription), GPT (GPT-4o) for containment analysis, web crawling, DM fit scoring, and playbook generation.
-   **Make.com**: Integrated for scenario auditing and blueprint management.
-   **Apollo.io**: Used for decision-maker enrichment and data acquisition.
-   **Outscraper**: Utilized for Google Maps searches and website lookup services.
-   **HubSpot**: Per-client OAuth integration. Auto-syncs call outcomes as notes/engagements, DMs as contacts, qualified deals as HubSpot deals, and companies. Won deals auto-generate invoices (sales proposals) as structured notes linked to deals/companies/contacts. HubSpot sync on proposals is optional — runs only if connected. Sync module: `server/hubspot-sync.ts`. OAuth routes: `server/hubspot.ts`. Token storage: `hubspot_tokens` DB table.
-   **Email Proposals**: Proposals from both My Leads and Pipeline pages are sent as formatted HTML emails to the client contact via the operator's SMTP settings (configured in Email Settings). Uses `sendProposalEmail` in `server/email-service.ts`. All user-supplied text is HTML-escaped to prevent injection.
-   **Twilio**: Connected via Replit integration (connector). Provides click-to-call, SMS, **automatic call recording with AI intelligence pipeline**, and **Real-Time Call Coaching (Phase 1)**. Service module: `server/twilio-service.ts`. Routes: `server/twilio-routes.ts`. Coaching module: `server/realtime-coaching.ts`. Every outbound Twilio call is automatically recorded (dual-channel). When recording completes, Twilio webhook triggers the full AI pipeline: download recording → Whisper transcription → deterministic containment analysis → GPT containment analysis → authority detection → follow-up date extraction → PostgreSQL storage (`twilio_recordings` table) → Airtable Calls writeback → EventBus `CALL_ANALYSIS_COMPLETE` event (feeds Sales Learning System: script evolution, pattern engine, observation engine). **Real-Time Coaching**: When a Twilio call starts from Call Mode, live audio is streamed via Twilio Media Streams (WebSocket at `/media-stream`) → OpenAI Realtime API (gpt-4o-realtime-preview, Whisper transcription, server VAD) → deterministic pattern detection (containment phrases, authority redirects, follow-up dates) → SSE push to frontend coaching panel. The Live Coach panel slides in from the right during active calls, showing: live scrolling transcript, color-coded coaching alerts (red=gatekeeper deflection, amber=authority redirect, blue=follow-up capture), and pinned talking points from web intel. After call ends, full transcript feeds back into the existing post-call analysis pipeline. **Call Status SSE (Phase 5)**: Twilio status callbacks (`POST /api/twilio/webhook/status`) receive call state changes (initiated/ringing/answered/completed/failed/busy/no-answer/canceled) and broadcast via SSE to `GET /api/twilio/call-status-stream/:sid`. Focus Mode subscribes to SSE first with polling fallback at 5s intervals — polling stops only after first real SSE `call_status` event arrives. **Recording Playback in Timeline (Phase 6)**: `GET /api/twilio/recording-by-company/:companyName` returns recordings for a company. Company Detail timeline enriches call/phone channel attempts with expandable RecordingCard showing duration, AI analysis, transcript, problem flags, authority status, suggested role, and follow-up date. Recording matching uses closest-time-within-1-hour heuristic. API endpoints: `GET /api/twilio/status`, `POST /api/twilio/call`, `POST /api/twilio/sms`, `GET /api/twilio/calls`, `GET /api/twilio/messages`, `GET /api/twilio/call/:sid`, `POST /api/twilio/webhook/recording` (Twilio callback), `POST /api/twilio/webhook/status` (Twilio status callback), `GET /api/twilio/call-status-stream/:sid` (SSE), `GET /api/twilio/recording-by-company/:companyName`, `GET /api/twilio/recordings`, `GET /api/twilio/recordings/:id`, `GET /api/twilio/coaching/:callSid` (SSE coaching stream), `GET /api/twilio/coaching-sessions` (active sessions). Buttons appear in Call Mode only when Twilio is connected. SMS consent page at `/sms-consent` for Twilio A2P verification.
-   **PostgreSQL**: Stores webhook logs, user accounts, client registry, and various system-specific data tables.
